# 3.1–3.6 — Log Management Infrastructure

> **Core Idea:** Log management infrastructure is the technology stack that collects, transports, stores, and analyzes logs. This chapter covers the architecture of log systems, the syslog protocol (the most widely used standard), SIEM platforms that provide correlation and intelligence, and other specialized log management tools.

---

## 🏗️ 3.1 — Architecture

### Components of a Log Management Infrastructure

```
┌─────────────┐  ┌─────────────┐  ┌─────────────┐
│  Log Source  │  │  Log Source  │  │  Log Source  │
│  (Firewall)  │  │  (Server OS) │  │ (Application)│
└──────┬──────┘  └──────┬──────┘  └──────┬──────┘
       │                │                │
       └────────────────┼────────────────┘
                        ↓
              ┌──────────────────┐
              │   LOG COLLECTOR  │  ← Agents, forwarders, syslog daemons
              │   / AGGREGATOR   │
              └────────┬─────────┘
                       ↓
              ┌──────────────────┐
              │   CENTRAL LOG    │  ← Syslog server, SIEM, data lake
              │     SERVER       │
              │  (Normalization, │
              │   Storage,       │
              │   Indexing)      │
              └────────┬─────────┘
                       ↓
              ┌──────────────────┐
              │    ANALYSIS &    │  ← Correlation rules, dashboards,
              │    MONITORING    │     alerting, reporting
              └──────────────────┘
```

### Architectural Tiers

| Tier | Function | Examples |
|------|----------|---------|
| **Generation** | Systems that create log events | Firewalls, servers, applications, endpoints |
| **Collection & Aggregation** | Gathering logs from sources and forwarding to central infrastructure | Syslog daemons, log forwarders (Fluentd, Logstash, rsyslog), agents (Splunk forwarder, Elastic Agent) |
| **Transport** | Moving log data from sources to central storage | Syslog (UDP/TCP), TLS-encrypted syslog, HTTP/S, message queues (Kafka) |
| **Storage** | Persisting log data for retention and analysis | Flat files, databases, indexed stores (Elasticsearch), WORM storage |
| **Analysis & Presentation** | Processing, correlating, visualizing, and alerting on log data | SIEM platforms, dashboards (Kibana, Grafana), correlation engines |

### Key Architectural Decisions

| Decision | Options | Consideration |
|----------|---------|--------------|
| **Push vs. Pull** | Push: sources send logs to server; Pull: server retrieves logs from sources | Push is more common and real-time; Pull gives more control but adds latency |
| **Agent vs. Agentless** | Agent: software installed on source; Agentless: uses native protocols (syslog, WMI) | Agents provide richer data but require deployment/management; agentless is lighter but may miss some data |
| **On-premise vs. Cloud** | Local infrastructure vs. cloud-based SIEM (Splunk Cloud, Azure Sentinel, Google Chronicle) | Cloud scales easily but raises data jurisdiction and cost concerns |
| **Filtering at source vs. central** | Filter noise at the source or collect everything centrally | Filtering at source reduces bandwidth/storage but may discard data needed later |

## ⚙️ 3.2 — Functions

### Core Functions of Log Management Infrastructure

| Function | Description | Why It Matters |
|----------|-------------|---------------|
| **Collection** | Gathering log data from all sources | Can't analyze what you don't collect |
| **Aggregation** | Combining logs from multiple sources into a central repository | Enables cross-source correlation |
| **Normalization** | Converting different log formats into a consistent, common format | Logs from Windows, Linux, firewalls all use different formats — normalization enables unified analysis |
| **Indexing** | Creating searchable indexes for fast querying | Analysts need to search billions of records in seconds |
| **Correlation** | Linking related events across multiple sources to identify patterns | A single attack generates events on many systems — correlation connects them |
| **Alerting** | Generating notifications when specific conditions or thresholds are met | Enables real-time detection of threats |
| **Reporting** | Generating regular and ad-hoc reports for management and compliance | Proves compliance; shows security posture trends |
| **Retention** | Storing logs for required periods with appropriate lifecycle management | Compliance mandates; forensic needs |
| **Disposal** | Securely deleting logs when retention period expires | Privacy and storage management |

### Log Normalization — Why It's Critical

Without normalization, the same event looks different across systems:

| Source | Raw Log Entry |
|--------|--------------|
| **Windows** | `Event ID 4625: An account failed to log on. Account: admin, Source: 192.168.1.50` |
| **Linux** | `Apr 14 03:15:22 server01 sshd[12345]: Failed password for admin from 192.168.1.50 port 22` |
| **Firewall** | `2026-04-14T03:15:22Z DENY user=admin src=192.168.1.50 dst=10.0.0.1 service=SSH` |

All three describe the **same event** (failed admin login from 192.168.1.50), but in completely different formats. Normalization converts all three into a common schema so they can be analyzed together.

## 📡 3.3 — Syslog-Based Centralized Logging Software

### What Is Syslog?

**Syslog** is the most widely used standard for log message transport. Originally defined in **RFC 3164** (BSD Syslog) and later standardized in **RFC 5424** (the modern syslog protocol).

### 3.3.1 — Syslog Format

#### Syslog Message Components

| Component          | Description                                                            | Example                                                    |
| ------------------ | ---------------------------------------------------------------------- | ---------------------------------------------------------- |
| **Priority (PRI)** | Calculated from Facility and Severity: PRI = (Facility × 8) + Severity | `<34>` = Facility 4 (auth) × 8 + Severity 2 (critical)     |
| **Header**         | Timestamp and hostname                                                 | `Apr 14 03:15:22 webserver01`                              |
| **Message (MSG)**  | The actual log content — tag (program name) and content                | `sshd[12345]: Failed password for admin from 192.168.1.50` |

#### Syslog Severity Levels (Memorize These!)

| Code | Severity | Description | Example |
|------|----------|-------------|---------|
| **0** | **Emergency** | System is unusable | Kernel panic, complete system failure |
| **1** | **Alert** | Immediate action needed | Database corruption, loss of primary ISP |
| **2** | **Critical** | Critical conditions | Hardware failure, application crash |
| **3** | **Error** | Error conditions | Failed to write to disk, connection refused |
| **4** | **Warning** | Warning conditions | Disk space at 90%, certificate expiring soon |
| **5** | **Notice** | Normal but significant | User login from unusual location, service restart |
| **6** | **Informational** | General operational messages | System startup, configuration change applied |
| **7** | **Debug** | Detailed debugging information | Variable values, step-by-step execution trace |

> **Mnemonic:** "**E**very **A**lert **C**ounts **E**ven **W**hen **N**obody **I**s **D**ebugging" (Emergency, Alert, Critical, Error, Warning, Notice, Informational, Debug)

#### Syslog Facility Codes (Key Ones)

| Code | Facility | Description |
|------|----------|-------------|
| 0 | kern | Kernel messages |
| 1 | user | User-level messages |
| 2 | mail | Mail system |
| 3 | daemon | System daemons |
| 4 | auth | Security/authentication messages |
| 10 | authpriv | Private authentication messages |
| 16–23 | local0–local7 | Locally defined / custom use |

### 3.3.2 — Syslog Security

#### Security Weaknesses of Traditional Syslog

| Weakness | Risk | Mitigation |
|----------|------|-----------|
| **UDP transport (port 514)** | Unreliable — no delivery confirmation; packets can be lost | Use TCP syslog (port 514 or 6514) or RELP for reliable delivery |
| **No encryption** | Log messages visible to anyone on the network (plaintext) | Use **TLS-encrypted syslog** (RFC 5425) on port 6514 |
| **No authentication** | Anyone can send fake syslog messages to the server (spoofing) | Use TLS with mutual certificate authentication |
| **No integrity protection** | Messages can be modified in transit | TLS provides integrity; also consider digital signatures |
| **Simple format** | Limited metadata; no structured data in RFC 3164 | Use **RFC 5424** format with structured data elements |

#### Secure Syslog Best Practices
- Use **TCP** (not UDP) for reliable delivery
- Use **TLS encryption** (port 6514) for confidentiality and integrity
- Implement **mutual authentication** with certificates
- Use **RFC 5424** (modern syslog) instead of RFC 3164 (legacy BSD syslog)
- Consider **RELP (Reliable Event Logging Protocol)** for guaranteed delivery
- **Rate limiting** to prevent log flooding denial-of-service attacks
- **Access controls** on the syslog server itself

#### Common Syslog Implementations

| Software | Description |
|----------|-------------|
| **rsyslog** | Enhanced syslog daemon for Linux; supports TCP, TLS, filtering, templates; most common on modern Linux |
| **syslog-ng** | Commercial/open-source alternative; powerful filtering, content-based routing, TLS support |
| **Windows Event Forwarding (WEF)** | Windows native log forwarding; not syslog but serves similar purpose |
| **Fluentd / Fluent Bit** | Modern log collectors that can receive syslog and forward to many backends |

## 🖥️ 3.4 — Security Information and Event Management (SIEM) Software

### What Is SIEM?

**SIEM** combines two formerly separate functions:
- **SIM (Security Information Management)** — long-term storage, analysis, and reporting of log data
- **SEM (Security Event Management)** — real-time monitoring, event correlation, and alerting

### Core SIEM Capabilities

| Capability | Description |
|-----------|-------------|
| **Log collection and aggregation** | Collects logs from all sources across the enterprise |
| **Normalization** | Converts all log formats into a common schema |
| **Real-time correlation** | Applies rules to detect multi-step attacks and complex patterns across sources |
| **Alerting and notification** | Generates alerts when correlation rules match or thresholds are exceeded |
| **Dashboards and visualization** | Real-time visual display of security status, trends, and metrics |
| **Incident management** | Workflow for tracking and managing security incidents |
| **Compliance reporting** | Pre-built and custom reports for regulatory requirements (PCI, HIPAA, SOX) |
| **Forensic investigation** | Powerful search and drill-down into historical log data |
| **Threat intelligence integration** | Correlates events with external threat feeds (known malicious IPs, IOCs) |
| **User and Entity Behavior Analytics (UEBA)** | Machine learning to detect anomalous user/entity behavior |

### How SIEM Correlation Works — Example

A single event might be harmless, but a pattern indicates an attack:

| Event # | Source | Event | Alone? |
|---------|--------|-------|--------|
| 1 | Firewall | Inbound connection from suspicious IP | Suspicious but common |
| 2 | IDS | SQL injection attempt detected | Happens frequently — could be a scanner |
| 3 | Web server | Error 500 on login page | Could be a bug |
| 4 | Database | Unusual query: "SELECT * FROM users" | Could be legitimate |
| 5 | DLP | Large data transfer to external IP | Could be a backup |

**Individually**: each event could be benign. **Correlated by SIEM**: Events 1→2→3→4→5 from the same source IP within 10 minutes = a **successful SQL injection attack with data exfiltration**. The SIEM generates a critical alert.

### Major SIEM Products

| Product | Type |
|---------|------|
| **Splunk** | Commercial leader; powerful search (SPL language); on-prem and cloud |
| **IBM QRadar** | Enterprise SIEM with strong correlation engine |
| **Microsoft Sentinel** | Cloud-native SIEM built on Azure; strong Microsoft ecosystem integration |
| **Elastic Security** | Open-source based (ELK stack); flexible and scalable |
| **Google Chronicle** | Cloud-native SIEM with Google-scale infrastructure |
| **LogRhythm** | SIEM with built-in SOAR (Security Orchestration, Automation, Response) |

## 📦 3.5 — Additional Types of Log Management Software

Beyond SIEM, other tools serve specific log management needs:

| Tool Type | Function | Example |
|-----------|----------|---------|
| **Log forwarders / Collectors** | Lightweight agents that collect and forward logs | Fluentd, Fluent Bit, NXLog, Logstash |
| **Log aggregators** | Centralize logs from many sources before sending to analysis | Kafka (message queue), rsyslog as relay |
| **Search and analytics platforms** | Index and search large volumes of log data | Elasticsearch + Kibana (ELK), OpenSearch |
| **Network forensics tools** | Capture and analyze network traffic (not just logs) | Wireshark, Zeek (Bro), NetworkMiner |
| **SOAR** (Security Orchestration, Automation, Response) | Automate incident response workflows based on log-triggered alerts | Palo Alto XSOAR, Splunk SOAR, IBM Resilient |
| **Cloud-native logging** | Built-in logging for cloud platforms | AWS CloudWatch, Azure Monitor, Google Cloud Logging |

## 💡 Key Takeaways

> [!tip] Remember
> - **Syslog** is the standard log protocol; severity levels 0–7 (Emergency to Debug) — **memorize them**
> - Traditional syslog (UDP, plaintext) is **insecure** — use **TLS-encrypted TCP syslog** (port 6514)
> - **SIEM = SIM + SEM** — combines storage/reporting with real-time correlation/alerting
> - **Normalization** is essential — different sources use different formats; you must convert to a common schema
> - **Correlation** is the SIEM's superpower — individual events may be harmless, but patterns reveal attacks
> - The syslog **PRI value** = (Facility × 8) + Severity
> - **RELP** provides reliable log transport; **WORM** storage provides integrity for legal admissibility
